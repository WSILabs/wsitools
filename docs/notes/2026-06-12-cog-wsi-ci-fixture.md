# CI fixture: `cog-wsi/CMU-1-Small-Region_cog-wsi.tiff`

> Provenance + activation note for the COG-WSI CI fixture added to
> `.github/fixtures.sha256`.

## What

A small COG-WSI file (~1.9 MB) added to the `wsilabs/wsi-fixtures` pool so CI can
exercise the COG-WSI **read** path (and the faithful associated-image round-trip)
without a local fixture pool.

- **Path (extracted):** `cog-wsi/CMU-1-Small-Region_cog-wsi.tiff`
- **SHA-256:** `c2f6268521b2e1c2c831d3ad6da8dfa309515f1039f61b73f9818bd998ae5f1e`
- **Size:** ~1.9 MB

## Why

- **opentile-go can't self-generate one.** opentile-go is a *reader*; testing its
  dedicated COG-WSI reader + associated-image decode in CI needs a real COG-WSI
  file. Its `TestAssociatedSourceRoundtrip` already references a cog-wsi fixture
  but only runs against a local `OPENTILE_TESTDIR` pool today.
- **Cross-repo regression guard for wsitools#1.** This file carries a **faithful
  LZW label** (387×463, Predictor 2, 67 strips) plus JPEG thumbnail + JPEG
  overview. It pins the interaction that produced wsitools#1 — wsitools writes it
  correctly ⇄ opentile-go reads it correctly. (wsitools' own faithful-copy CI
  test already covers write+read by converting the in-CI SVS → cog-wsi in a temp
  dir; this committed golden adds a *stable read* fixture and the opentile-go-side
  coverage.)

## How it was generated

```sh
# from the fixture pool root, with a wsitools build that has the
# faithful associated-copy fix (>= main 6eeb480, v0.23.0-dev):
wsitools convert --to cog-wsi \
  -o cog-wsi/CMU-1-Small-Region_cog-wsi.tiff \
  svs/CMU-1-Small-Region.svs
```

- **Source:** `svs/CMU-1-Small-Region.svs` (already in the fixture pool).
- **Generated by:** wsitools `v0.23.0-dev` (main `6eeb480`, the merge of
  `feat/associated-faithful-copy` — opentile-go v0.39.0 `AssociatedSourceOf`).
- **Contents:** one pyramid level (JPEG tiles) + faithful associated `label`
  (LZW + Predictor 2), `thumbnail` (JPEG), `overview` (JPEG).

## Determinism note

The `cogwsiwriter` tile-ordering pass is **nondeterministic** (see the
`pipeline-nondeterminism` rule), so regenerating this file will NOT reproduce the
same bytes / SHA — but it *will* be pixel-identical. The committed artifact is the
canonical golden; `fixtures.sha256` pins its exact bytes. To refresh it:
regenerate, recompute the SHA, re-upload the tar, and update `fixtures.sha256`.

## CI wiring / activation

CI (`.github/workflows/ci.yml`) downloads every `*.tar` asset from the
`wsilabs/wsi-fixtures` release, extracts it, and verifies each file against
`.github/fixtures.sha256`. The cache key hashes `fixtures.sha256`, so adding the
line above auto-invalidates the cache and forces a re-download.

**To activate (do this before merging this branch, or CI's SHA-verify step will
fail with a missing file):**

1. Upload `cog-wsi.tar` (extracts to `cog-wsi/CMU-1-Small-Region_cog-wsi.tiff`,
   SHA above) to the `wsilabs/wsi-fixtures` release.
   - **Amend the existing `v1` release** (simplest): the `gh release download v1
     --pattern '*.tar'` step picks up the new tar automatically — **no `ci.yml`
     change needed**.
   - **Or cut a new `v2` release:** then also bump `ci.yml` (`gh release download
     v1` → `v2`) and the cache-key prefix (`wsi-fixtures-v1-` → `wsi-fixtures-v2-`).
2. Merge this branch (`chore/ci-cog-wsi-fixture`).
3. Optionally, in opentile-go: wire the same fixture + flip its known-corrupt /
   local-only cog-wsi associated assertion to a CI assertion (its reader is the
   primary beneficiary).

The `cog-wsi.tar` for upload is built with:
```sh
tar -C <fixture-pool> -cf cog-wsi.tar cog-wsi/CMU-1-Small-Region_cog-wsi.tiff
```
