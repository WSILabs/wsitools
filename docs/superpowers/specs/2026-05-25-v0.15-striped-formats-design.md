# wsitools v0.15 — Striped-Format Source Support (NDPI / OME-OneFrame / Leica-SCN)

**Status:** Design approved 2026-05-25.

## 1. Goal

Remove wsitools' stale "v0.2 sanity gate" that rejects NDPI,
OME-OneFrame, and single-image Leica-SCN slides. opentile-go's
readers for all three formats already synthesize tile geometry
(verified — every format reports populated `Size` / `TileSize` /
`Grid` and returns valid JPEG bytes from `RawTile(0, 0, 0)`); the
wsitools source layer just needs to trust the synthesis.

After v0.15, every CLI subcommand (info, transcode, downsample,
convert, hash, extract, dump-ifds, region) works on these three
striped formats with no special-casing.

## 2. Scope

In scope for v0.15:

- Drop the per-format gates in `internal/source/opentile.go` for
  `FormatNDPI`, `FormatLeicaSCN`, and the OME-OneFrame `TileSize==0`
  rejection.
- Update `ErrUnsupportedFormat`'s message to drop the "v0.2" version
  marker and the format enumeration; it stays in the package for
  genuinely-unsupported future formats.
- Replace the format-rejection tests in
  `internal/source/opentile_test.go` with format-acceptance tests
  that verify the slide opens cleanly.
- New integration tests covering info / transcode / downsample /
  hash / convert / extract on each format's sample fixtures.
- Documentation updates:
  - `docs/roadmap.md` — remove "Deferred from v0.2" entries for the
    three formats; add a v0.15 entry to the shipped list.
  - `README.md` (if it lists supported formats) — add the three.
- CHANGELOG.md v0.15.0 entry documenting the new format support,
  the convert "bit-exact tile copy" caveat for striped sources,
  and the deferred multi-channel SCN + multi-image OME work.

Out of scope (deferred):

- Multi-channel fluorescence Leica SCN
  (Leica-Fluorescence-1.scn). wsitools' transcode/downsample
  pipeline assumes RGB; non-RGB channel handling is a future
  release.
- Multi-image OME-TIFF where multiple `<Image>` series each carry
  their own pyramid. `info` currently shows image 0 only; the
  multi-image surface is a separate piece of work.
- Changes to opentile-go. v0.25.0 already exposes the synthesized
  tile geometry; no opentile-go release required.
- Goldens-byte-stable expansion to include the three new formats.
  The byte-stability check is per-binary-version, not per-format;
  adding formats doesn't change the determinism guarantee.

Non-goals:

- Performance optimization of the synthesis path.
- Output format changes (transcode still re-encodes; convert still
  copies tile bytes — including the synthesized ones for striped
  sources).

## 4. The bit-exact tile-copy caveat

`convert --to cog-wsi` historically promised bit-exact tile-copy
from source to output — the compressed tile bytes in the source
file appear verbatim in the destination COG-WSI file. This is the
basis of the v0.6 convert spec.

For striped sources (NDPI, OME-OneFrame), the source file has NO
tile bytes — it has strip bytes. opentile-go synthesizes JPEG
tile bytes on demand by:

1. Identifying the strip(s) containing the requested tile region.
2. Extracting the MCU-aligned sub-region (NDPI) or stitching a
   strip-row + JPEG-header (OME-OneFrame).
3. Emitting a valid standalone JPEG.

The synthesis is **deterministic** (same input → same output, run
to run), but the resulting tile bytes are NOT the source's
on-disk bytes. They're synthesized bytes that happen to decode
to the same pixels as the corresponding source region.

**v0.15 documents this clearly:**

- `convert` retains its bit-exact-tile-copy promise FOR
  natively-tiled sources (SVS, Philips, OME-tiled, BIF, IFE,
  generic-TIFF, COG-WSI, SZI).
- For striped sources (NDPI, OME-OneFrame), the COG-WSI output is
  reproducible byte-for-byte from the same source — but the bytes
  in the output are opentile-go's synthesized JPEG output, not the
  source's strip bytes.
- For natively-tiled Leica-SCN, the bit-exact-tile-copy promise
  applies as normal.

`convert --help` text and the v0.6 convert design's "bit-exact
promise" section get one additional paragraph documenting this.

## 5. Test plan

### 5.1 Source-layer unit tests

Replace `TestOpen_NDPI_Rejects` and `TestOpen_LeicaSCN_Rejects`
with `TestOpen_NDPI_Accepts` and `TestOpen_LeicaSCN_Accepts`. New
test `TestOpen_OMEOneFrame_Accepts`. Each opens a sample file via
`source.Open` and verifies:

- No error returned.
- `source.Source` has at least one Level.
- Level 0's Size / TileSize / Grid are populated (non-zero).

Sample paths (gated by `WSI_TOOLS_TESTDIR`):

- NDPI: `ndpi/CMU-1.ndpi` (or `OS-2.ndpi`)
- Leica SCN: `scn/Leica-1.scn`
- OME-OneFrame: a OneFrame OME-TIFF if available; if no fixture,
  skip with a documented note (the gate-removal still ships even
  without a fixture).

### 5.2 CLI integration tests

For each of the three formats, exercise the CLI commands via
`exec.Command`:

- `info` — verify exit 0 + non-empty output.
- `transcode --codec jpeg --container svs` — verify exit 0 +
  output file exists + decodes as TIFF.
- `downsample --factor 2` — verify exit 0 + output is a valid SVS.
- `hash --mode pixel` — verify exit 0 + reasonable hash length.
- `convert --to cog-wsi` — verify exit 0 + output is a valid
  COG-WSI file. The "bit-exact tile copy" expectation does NOT
  apply for striped sources; the test should NOT compare tile
  bytes to the source.
- `extract --label` or `--macro` — verify exit 0 + PNG output if
  the sample has the associated image.

These tests gate on `WSI_TOOLS_TESTDIR` and skip cleanly if
samples are missing.

### 5.3 Acceptance gate

`make goldens-byte-stable` continues to pass on the existing
SVS-only sample set. The three new formats don't need byte-stable
goldens (transcode determinism is per-version; running on new
formats doesn't change the existing guarantee).

The `tests/parity/` integration suite (if present in wsitools)
may want NDPI / SCN / OME-OneFrame additions in a follow-up; not
required for v0.15.

## 6. CHANGELOG v0.15.0 entry

```markdown
## [0.15.0] — TBD

### Added

- NDPI, OME-OneFrame, and Leica SCN slides are now supported
  across all CLI subcommands (info, transcode, downsample,
  convert, hash, extract, dump-ifds, region). opentile-go
  synthesizes tile geometry from striped MCU streams (NDPI) and
  single-frame OME (OME-OneFrame); wsitools' tile-pipeline now
  operates on opentile-go-tiled output verbatim.

### Changed

- Dropped the "v0.2 sanity gate" in `internal/source/opentile.go`
  that rejected NDPI / OME-OneFrame / Leica-SCN. Stale since
  opentile-go v0.14+ began synthesizing tile geometry.
- `ErrUnsupportedFormat`'s message updated to drop the "v0.2"
  version marker; the sentinel remains for genuinely-unsupported
  future formats.

### Bit-exact tile-copy caveat (convert)

`convert --to cog-wsi` from natively-tiled sources (SVS, Philips,
OME-tiled, BIF, IFE, generic-TIFF, COG-WSI, SZI, single-image
SCN) continues to produce bit-exact tile-copy COG-WSI output —
the source's compressed tile bytes appear verbatim in the
destination.

From striped sources (NDPI, OME-OneFrame), the COG-WSI output
contains opentile-go's synthesized JPEG tile bytes. These bytes
decode to the same pixels as the source region and are
deterministic (same input → same output), but they are NOT the
source's on-disk bytes (NDPI / OneFrame source files don't carry
tile bytes — they carry strip bytes).

### Out of scope (deferred)

- Multi-channel fluorescence Leica SCN
  (Leica-Fluorescence-1.scn). transcode/downsample assume RGB
  channels; multi-channel handling is a future release.
- Multi-image OME-TIFF where multiple `<Image>` series each carry
  their own pyramid. `info` shows image 0 only.
```

Single wsitools release. opentile-go untouched. Cut as v0.15.0
from `main` after merge. Branch: `feat/v15-striped-formats`.
