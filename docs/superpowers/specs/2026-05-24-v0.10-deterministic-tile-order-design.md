# v0.10 — Deterministic Tile Order

**Status:** Design approved 2026-05-24.

## 1. Goal

Make wsitools' `transcode`, `downsample`, and `convert` commands produce
**byte-deterministic** output: the same input slide and CLI invocation produce
the same output bytes regardless of run, CPU count, or scheduler whims.
Today the parallel tile-writer emits tiles in worker-arrival order, recording
tile offsets at write-time, so file-level SHA-256 of the output varies
run-to-run (verified empirically during v0.9.0 port — three runs of the same
v0.8.1 binary on `CMU-1-Small-Region.svs` produced three different hashes).

Determinism enables: reproducible builds, content-addressable storage (S3
dedup, IPFS), reliable file-SHA goldens for regression testing, and a clean
hook for future format-locality optimizations (Hilbert/Morton ordering for
COG-WSI cloud range-reads; Iris-format-specific super-block layouts).

The cost is essentially nil: the existing parallel encoder pipeline is
CPU-bound on JPEG/JPEG2000 encode, not on the sequential write step. Inserting
a small reorder buffer between worker channel and disk-write adds ~1–2 MB of
memory and ~0% throughput hit, in exchange for the byte-stability guarantee.

## 2. Scope

In scope for v0.10:

- New pure-Go package `internal/tiff/tileorder` defining the `OrderStrategy`
  interface plus shipped strategies: `RowMajor`, `HilbertCurve`, `Morton`.
- Bounded reorder buffer in `internal/tiff/streamwriter` between the parallel
  worker channel and the actual file-write call. Back-pressures workers when
  full.
- `internal/tiff/cogwsiwriter` finalize pass walks the spooled tile manifest
  in `strategy.Index` order instead of naive raster order.
- `--tile-order={row-major|hilbert|morton}` CLI flag on `transcode`,
  `downsample`, `convert`. Default `row-major`. Format-validated per writer.
- Per-format `AcceptsOrder` predicate on each writer variant (SVS strict
  row-major; COG-WSI all three; generic-TIFF and OME-TIFF permissive).
- New byte-stability test infrastructure: run the same invocation N times at
  varying `GOMAXPROCS`, assert identical SHAs.
- New canonical goldens file `docs/superpowers/golden-masters-v0.10.0.txt`
  documenting per (format × codec × sample) hashes. Replaces the busted
  v0.6/v0.9 golden methodology.

Out of scope (parked for later releases):

- Iris writer. When that lands, it declares its own `AcceptsOrder` set and
  optionally registers Iris-specific strategies. No other changes required.
- Cross-level emission order. Level emission order is format-mandated (SVS:
  largest-first; COG-WSI: overview-first; generic: any). Today it's
  caller-driven; v0.10 doesn't touch this.
- `--tile-order` defaults that vary by container. v0.10 defaults to
  `row-major` everywhere. Container-aware smart defaults are a future tweak.
- Unordered/legacy emission mode. Determinism is a guarantee, not an opt-in.
  The buffer is so cheap there's no performance argument for bypassing it.
- Reorder-buffer capacity tuning. Internal Go-API option only
  (`WithReorderCapacity`), no CLI exposure, no public documentation beyond a
  code comment.

Dependency direction: `tileorder` is a pure-Go leaf package. Both writers
import it; the CLI imports it for the `ByName` lookup. No sibling-to-sibling
coupling between streamwriter and cogwsiwriter.

## 4. The `tileorder` package

### 4.1 `OrderStrategy` interface

```go
package tileorder

// OrderStrategy maps tile grid coordinates to a linear emission index.
// Implementations must be:
//   - bijective on the [0, tilesX) × [0, tilesY) domain
//   - deterministic across calls for the same (x, y, tilesX, tilesY)
//   - safe for concurrent calls (Submit happens from N worker goroutines)
//
// Strategies MAY perform per-(tilesX, tilesY) memoization (sync.Map keyed
// by grid shape) — required for non-power-of-2 grids with Hilbert and
// Morton orderings, which precompute a rank table. RowMajor is pure-
// function; Hilbert/Morton hold a small cache. Memoization is one-time
// per unique grid shape (typically once per pyramid level), amortized
// over O(tilesX * tilesY) calls.
//
// All strategies operate within a single level. Cross-level ordering is
// format-mandated and managed by the caller's loop over levels.
type OrderStrategy interface {
    // Name returns the canonical lowercase identifier ("row-major",
    // "hilbert", "morton"). Used for ByName lookup and error messages.
    Name() string

    // Index returns the linear emission index for tile (x, y) in a
    // tilesX × tilesY grid. Result is in [0, tilesX*tilesY).
    Index(x, y, tilesX, tilesY uint32) uint32

    // IndexToXY is the inverse of Index. Used by cogwsiwriter's finalize
    // pass to iterate tiles in emission order without storing the
    // forward mapping.
    IndexToXY(idx, tilesX, tilesY uint32) (x, y uint32)
}
```

### 4.2 Shipped strategies

- **`RowMajor`** — `Index = y*tilesX + x`. Universal default. Always accepted
  by every writer.
- **`HilbertCurve`** — Hilbert space-filling curve, computed via the standard
  iterative bit-twiddle algorithm. Better 2-D locality for cloud range-reads
  on COG-WSI. For non-power-of-2 grids: compute Hilbert indices in the
  smallest power-of-2 grid containing W×H, collect only points within W×H,
  sort by Hilbert index, assign emission ranks 0..W*H-1. This rank table is
  memoized per `(tilesX, tilesY)` shape via a package-level `sync.Map` so the
  precomputation cost (O(W*H log(W*H))) is paid once per pyramid level.
- **`Morton`** — Morton/Z-order curve via bit-interleaving. Same
  non-power-of-2 handling as HilbertCurve (rank table memoized per grid
  shape). Cheaper to compute than Hilbert; slightly weaker 2-D locality.

### 4.3 Registry

```go
// ByName returns the strategy with the given canonical name, or an error
// for unknown names. Used by the CLI to translate --tile-order=... into
// a strategy instance.
func ByName(name string) (OrderStrategy, error)

// Known returns the names of all registered strategies. For help text.
func Known() []string
```

Registry is package-internal and uninstantiated — the three concrete strategies
are exported as package-level vars (`RowMajor`, `HilbertCurve`, `Morton`) and
the `ByName` function dispatches via a `map[string]OrderStrategy` initialized
in `init()`.

## 5. Reorder buffer

### 5.1 Per-level buffer

Each `LevelHandle` owns its own `reorderBuffer` instance. Constructed when
`AddLevel(...)` is called; destroyed when the level's last tile is emitted.

```go
type reorderBuffer struct {
    strategy   tileorder.OrderStrategy
    tilesX     uint32
    tilesY     uint32
    total      uint32

    mu       sync.Mutex
    cond     *sync.Cond
    pending  map[uint32][]byte
    nextIdx  uint32
    emitted  uint32
    capacity uint32
    err      error  // sticky abort error
}
```

### 5.2 `Submit` (workers)

```go
func (b *reorderBuffer) Submit(x, y uint32, compressed []byte) error {
    idx := b.strategy.Index(x, y, b.tilesX, b.tilesY)
    b.mu.Lock()
    defer b.mu.Unlock()
    for {
        if b.err != nil {
            return b.err
        }
        if uint32(len(b.pending)) < b.capacity || idx == b.nextIdx {
            break
        }
        b.cond.Wait()  // back-pressure: buffer full, this tile not at head
    }
    b.pending[idx] = compressed
    b.cond.Broadcast()
    return nil
}
```

A tile at `idx == nextIdx` is always admitted regardless of capacity, so the
buffer can never deadlock against itself.

### 5.3 `NextReady` (Sink)

```go
func (b *reorderBuffer) NextReady() (idx uint32, compressed []byte, ok bool, err error) {
    b.mu.Lock()
    defer b.mu.Unlock()
    for {
        if b.err != nil {
            return 0, nil, false, b.err
        }
        if b.emitted == b.total {
            return 0, nil, false, nil
        }
        if bytes, present := b.pending[b.nextIdx]; present {
            idx = b.nextIdx
            delete(b.pending, b.nextIdx)
            b.nextIdx++
            b.emitted++
            b.cond.Broadcast()
            return idx, bytes, true, nil
        }
        b.cond.Wait()
    }
}
```

### 5.4 `Abort`

```go
func (b *reorderBuffer) Abort(err error) {
    b.mu.Lock()
    defer b.mu.Unlock()
    if b.err == nil {
        b.err = err
    }
    b.cond.Broadcast()  // wake everyone
}
```

Called by the pipeline when any worker errors or context is cancelled. All
pending Submits and the Sink's NextReady return the error immediately.

### 5.5 Capacity

`capacity = max(8, 2 * numWorkers)`. 8 floor so single-CPU runs have minimum
slack; 2× workers heuristic gives one-tile-late variance budget. Configurable
via `WithReorderCapacity(int)` (Go-API option, no CLI surface). Above ~4×
workers gives diminishing returns; below ~workers/2 forces frequent
back-pressure.

### 5.6 Memory bound

Worst case `capacity × avg_compressed_tile_size`. For 32 workers × 60 KB JPEG
tiles ≈ 2 MB. Multi-GB source slides are unaffected.

## 6. Writer integration

### 6.1 streamwriter

`AddLevel` gains variadic options:

```go
type levelOption interface{ applyTo(*levelEntry) }

func WithLevelOrder(o tileorder.OrderStrategy) levelOption
func WithReorderCapacity(cap int) levelOption  // debug

func (w *Writer) AddLevel(ifd uint32, tilesX, tilesY uint32, ..., opts ...levelOption) (*LevelHandle, error) {
    le := &levelEntry{order: w.defaultOrder, capacity: 2 * w.numWorkers}
    for _, opt := range opts { opt.applyTo(le) }
    if !w.acceptsOrder(le.order) {
        return nil, fmt.Errorf("tile order %q not supported by writer format %q (allowed: %v)",
            le.order.Name(), w.formatName, w.acceptedOrderNames())
    }
    // construct LevelHandle with reorderBuffer(le.order, tilesX, tilesY, le.capacity)
}
```

`Writer` carries `formatName string` and `acceptedOrders []string` set at
construction. Variant constructors:

- `NewSVSWriter` — formatName="svs", acceptedOrders=["row-major"]
- `NewGenericTIFFWriter` — formatName="tiff", acceptedOrders=["row-major", "hilbert", "morton"]
- `NewOMETIFFWriter` — formatName="ome-tiff", acceptedOrders=["row-major", "hilbert", "morton"]

`WriteTile` signature is **unchanged**:

```go
func (h *LevelHandle) WriteTile(x, y uint32, compressed []byte) error {
    return h.buf.Submit(x, y, compressed)
}
```

A new internal method handles the actual append + offset recording:

```go
func (h *LevelHandle) writeTileAtIndex(idx uint32, compressed []byte) error {
    off := h.w.tell()
    if _, err := h.w.write(compressed); err != nil { return err }
    h.entry.tileOffsets[idx] = off
    h.entry.tileByteCounts[idx] = uint32(len(compressed))
    return nil
}
```

Called only from the Sink loop, never by workers.

### 6.2 cogwsiwriter

cogwsiwriter currently spools tiles to a temp file (variable compressed size
requires sizing before COG-WSI's early-IFD layout can be computed) and then,
during finalize, copies spooled bytes into the final file in raster order.

Change: replace the raster-order walk with strategy-order walk:

```go
// Before:
for y := uint32(0); y < tilesY; y++ {
    for x := uint32(0); x < tilesX; x++ {
        w.copyTileFromSpool(x, y)
    }
}

// After:
total := tilesX * tilesY
for idx := uint32(0); idx < total; idx++ {
    x, y := w.order.IndexToXY(idx, tilesX, tilesY)
    w.copyTileFromSpool(x, y)
}
```

cogwsiwriter's constructor (`NewCOGWSIWriter`) gets a `WithDefaultOrder(o)`
option, defaulting to `RowMajor`. `acceptedOrders` for COG-WSI: row-major,
hilbert, morton.

### 6.3 Sink (pipeline.go)

Currently the Sink reads tiles from the workers' output channel and calls
`lh.WriteTile` directly. With the reorder buffer, the Sink splits into two
goroutines per level:

```go
// Goroutine 1: workers → buffer adapter
go func() {
    for tile := range out {
        if err := lh.WriteTile(tile.X, tile.Y, tile.Bytes); err != nil {
            lh.Abort(err)
            return
        }
    }
    lh.CloseInput()
}()

// Goroutine 2: ordered drain to disk (runs in main pipeline goroutine)
for {
    idx, bytes, ok, err := lh.NextReady()
    if err != nil { return err }
    if !ok { break }
    if err := lh.writeTileAtIndex(idx, bytes); err != nil { return err }
}
```

Both goroutines share the LevelHandle's reorderBuffer; `Abort` propagates
errors cleanly.

## 8. Testing strategy

### 8.1 `tileorder` unit tests

For each shipped strategy:

- `Index(x,y,W,H)` produces a unique value in `[0, W*H)` for every input;
  exhaustive coverage on small grids (8×8, 16×16, 7×11, 1×1).
- `IndexToXY(Index(x,y,W,H), W, H) == (x,y)` — round-trip identity.
- `Name()` returns the expected lowercase string.
- `ByName("hilbert")` returns the strategy; `ByName("garbage")` returns a
  clear error including the list of `Known()` names.

Coverage target: ≥95% for the `tileorder` package.

### 8.2 `reorder` buffer tests

- Submit in reverse order → NextReady drains in strategy order.
- Submit in randomized order → NextReady drains in strategy order.
- Capacity bound: with `cap=4`, push 10 tiles in pathological order in
  goroutines; verify Submit blocks (channel + timeout sentinel) until Sink
  drains tile 0.
- Abort path: call `Abort(err)` mid-flight; verify pending Submits and
  NextReady all return that error.
- Level completion: `NextReady` returns `ok=false, err=nil` exactly when all
  `total` tiles have been emitted.

Coverage target: ≥90% for the buffer file (concurrency paths matter).

### 8.3 End-to-end byte-stability

New `make goldens-byte-stable` target (and CI step where sample files are
available):

```bash
for SAMPLE in $(ls "$WSI_TOOLS_TESTDIR"/svs/*.svs); do
    OUT1=$(mktemp).svs; OUT2=$(mktemp).svs; OUT3=$(mktemp).svs
    GOMAXPROCS=2  ./bin/wsitools transcode --codec jpeg --container svs -f -o $OUT1 $SAMPLE
    GOMAXPROCS=8  ./bin/wsitools transcode --codec jpeg --container svs -f -o $OUT2 $SAMPLE
    GOMAXPROCS=32 ./bin/wsitools transcode --codec jpeg --container svs -f -o $OUT3 $SAMPLE
    H1=$(shasum -a 256 $OUT1 | awk '{print $1}')
    H2=$(shasum -a 256 $OUT2 | awk '{print $1}')
    H3=$(shasum -a 256 $OUT3 | awk '{print $1}')
    [ "$H1" = "$H2" ] && [ "$H2" = "$H3" ] || { echo "byte-stability FAIL on $SAMPLE: $H1/$H2/$H3"; exit 1; }
done
```

### 8.4 Canonical goldens

Capture per (format × codec × sample) hashes into
`docs/superpowers/golden-masters-v0.10.0.txt` after v0.10 is built green.
Annotate the existing `golden-masters-v0.6.0-transcode.txt` as historical-only
(its hashes pre-date the determinism guarantee and aren't reproducible).

Future PRs that mutate output bytes must update the goldens file with a
justification in the PR description.

### 8.5 Format validation tests

Table-driven test across the (writer × strategy) matrix asserting accept/reject
behavior, e.g.:

| Writer        | row-major | hilbert | morton |
| ------------- | --------- | ------- | ------ |
| SVS           | accept    | reject  | reject |
| Generic-TIFF  | accept    | accept  | accept |
| OME-TIFF      | accept    | accept  | accept |
| COG-WSI       | accept    | accept  | accept |

CLI rejection path: `wsitools transcode --container svs --tile-order=hilbert ...`
exits nonzero with the expected error message; covered by an integration test
under `cmd/wsitools/`.

## 9. Migration & backward compatibility

### 9.1 User-facing

- Default behavior unchanged: unflagged invocations produce row-major tile
  layout, same as v0.9. Only difference is output is now byte-deterministic
  across runs.
- New optional `--tile-order` flag with sensible defaults and format
  restrictions.
- Pixel-equality with v0.9 preserved — only on-disk byte order changes, never
  decoded pixel values.
- All existing CLI invocations work identically.

### 9.2 Go API

- `streamwriter.LevelHandle.WriteTile(x, y, bytes)` signature unchanged.
- `streamwriter.AddLevel(...)` gains variadic `...levelOption` — additive,
  existing zero-option calls compile unchanged.
- New public types/funcs in `internal/tiff/tileorder` (all net-new).
- `cogwsiwriter` finalize internals change; external API unchanged.

### 9.3 Goldens file

Annotate `docs/superpowers/golden-masters-v0.6.0-transcode.txt` with a header
noting the v0.6/v0.9 hashes are historical artifacts captured from a
non-deterministic writer (not reproducible). Create
