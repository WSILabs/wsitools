# wsitools v0.19 — CI Fixture Pipeline + `wsi-fixtures` Repo

**Status:** Design approved 2026-05-28.

## 1. Goal

Wire wsitools CI to fetch a minimal set of WSI fixtures from a new
public-license fixtures repo, so the integration tests that currently
`t.Skip` when `WSI_TOOLS_TESTDIR` is unset start running on every
push and PR.

After v0.19:
- CI matrix runs the existing `TestConvertToDZI`, `TestConvertToSZI`,
  `TestConvertNDPIToDZI`, `TestConvertNDPIToSZI`, and friends.
- Regressions visible to all consumers in real CI matrix (incl.
  macOS / Linux / Windows) — would have caught the v0.17 Windows
  linker bug before tagging.
- The fixtures repo (`wsilabs/wsi-fixtures`) is reusable by
  opentile-go and future WSI tooling.

## 2. Scope

In scope for v0.19:

- **New repo `wsilabs/wsi-fixtures`.** Public, CC0/public-license
  fixture corpus.
- **Initial fixture set (two files, ~191 MB total):**
  - `CMU-1-Small-Region.svs` (1.85 MB, CC0; source: openslide-testdata
    Aperio set).
  - `CMU-1.ndpi` (188.86 MB, CC0; source: openslide-testdata Hamamatsu
    set).
- **Per-fixture provenance + license artifacts** alongside each
  binary.
- **Machine-readable `manifest.json`** indexing fixtures with
  SHA-256, license SPDX, source URL.
- **GitHub Release `v1`** hosts per-format tarballs (`svs.tar`,
  `ndpi.tar`). **Releases are immutable**: once `v1` is published,
  its assets never change. Any future fixture addition or change
  produces a new release (`v2`, `v3`, ...) that supersedes `v1`
  while leaving `v1` intact. Consumers pin to a specific release
  tag for reproducibility — a wsitools commit referencing `v1`
  always sees the same fixture bytes.
- **wsitools `.github/workflows/ci.yml` update** that caches +
  downloads + verifies fixtures before running tests.
- **`.github/fixtures.sha256`** in wsitools, pinning the expected
  hashes for cache-integrity verification.

Out of scope (deferred):

- **Per-format coverage beyond NDPI + SVS.** Philips, OME-TIFF, BIF,
  IFE, SCN, MRXS, DICOM, SZI, generic-TIFF, COG-WSI each require a
  separate licensing audit. Add incrementally as needed.
- **OS-2.ndpi (931 MB) for `make bench-dzi` CI gate.** Promoting
  the libvips parity benchmark to CI needs libvips in the runner
  image plus a flake-tolerant ratio threshold. Separate design.
- **Hamamatsu-1.ndpi (6.43 GB).** CC0 but exceeds GitHub's 2 GB
  per-file release asset limit. Future: chunked upload + assembly,
  or external host (R2/B2).
- **opentile-go CI fixture pipeline.** wsi-fixtures is designed to
  serve opentile-go's CI too; wiring it up belongs to a separate
  opentile-go release.
- **Writer subdirectory layout (`svs/aperio/...`).** v0.19's two
  fixtures have obvious single writers; subdir is noise until a
  format has multiple writers in the corpus. Added incrementally
  when needed; the CI workflow doesn't care about the path depth
  inside a format tarball.

Non-goals:

- **Private / PHI-bearing slides.** wsi-fixtures is intentionally
  public-license-only.
- **Fetch-from-upstream-URL fallback in CI.** wsi-fixtures hosts
  bytes; we don't introduce upstream-URL fallback paths that break
  when third-party CDNs change.
- **Git LFS.** Release assets are simpler and avoid bloating
  wsi-fixtures' git history.
- **gzipped tarballs.** Fixture binaries are already
  JPEG-compressed; gzip adds zero net benefit and decode cost.

## 3. Repo: `wsilabs/wsi-fixtures`

### 3.1 Layout

```
wsi-fixtures/
├── README.md
├── LICENSE                  # MIT for the metadata + scripts; per-fixture binaries
│                            # are governed by their own .LICENSE files.
├── manifest.json
├── scripts/
│   ├── verify.sh            # validates SHA-256 of every fixture against manifest
│   └── pack.sh              # builds the per-format tarballs uploaded to a release
└── fixtures/
    ├── svs/
    │   ├── CMU-1-Small-Region.svs
    │   ├── CMU-1-Small-Region.svs.LICENSE
    │   └── CMU-1-Small-Region.svs.PROVENANCE
    └── ndpi/
        ├── CMU-1.ndpi
        ├── CMU-1.ndpi.LICENSE
        └── CMU-1.ndpi.PROVENANCE
```

**`fixtures/<format>/<name>.LICENSE`** is the verbatim license text
or a one-line SPDX reference when license-spdx is sufficient
(e.g. `CC0-1.0` from the CC0 deed). For fixtures with bespoke terms
(some scanner vendors), the full terms text is captured.

**`fixtures/<format>/<name>.PROVENANCE`** captures:
- Source URL the fixture was downloaded from.
- Retrieval date.
- SHA-256 of the file (matches `manifest.json`).
- Citation: the upstream maintainer + any relevant paper/contribution.
- License SPDX identifier.

### 3.2 `manifest.json`

```json
{
  "version": "v1",
  "fixtures": [
    {
      "path": "fixtures/svs/CMU-1-Small-Region.svs",
      "size": 1938955,
      "sha256": "<computed at audit time>",
      "source_url": "https://openslide.cs.cmu.edu/download/openslide-testdata/Aperio/CMU-1-Small-Region.svs",
      "retrieval_date": "<audit date>",
      "license_spdx": "CC0-1.0",
      "license_file": "fixtures/svs/CMU-1-Small-Region.svs.LICENSE",
      "tarball_asset": "svs.tar"
    },
    {
      "path": "fixtures/ndpi/CMU-1.ndpi",
      "size": 198030965,
      "sha256": "<computed at audit time>",
      "source_url": "https://openslide.cs.cmu.edu/download/openslide-testdata/Hamamatsu/CMU-1.ndpi",
      "retrieval_date": "<audit date>",
      "license_spdx": "CC0-1.0",
      "license_file": "fixtures/ndpi/CMU-1.ndpi.LICENSE",
      "tarball_asset": "ndpi.tar"
    }
  ]
}
```

`tarball_asset` is the asset name in the GitHub Release that
contains this fixture. CI uses this to know which tarball to pull
when only a subset of formats is needed (e.g. an opentile-go
workflow that doesn't need DICOM).

### 3.3 Release assets

- **Releases are immutable.** `v1` is published once with the
  initial fixture set; the assets attached to `v1` never change.
- Future fixture additions or changes publish a new tag (`v2`,
  `v3`, ...). Old tags stay accessible forever; consumers upgrade
  by explicitly bumping the release reference.
- One tarball per format per release: `svs.tar`, `ndpi.tar` (in
  v0.19's `v1` release; more formats in later releases).
- Tarballs are uncompressed `.tar` (NOT `.tar.gz`). Fixture
  binaries are already JPEG-compressed; gzip adds zero size benefit
  + decode overhead.
- Each tarball preserves the in-repo path so `tar xf svs.tar`
  extracts to `svs/CMU-1-Small-Region.svs` (etc.).

### 3.4 README.md

Explains:
- What this repo is + who uses it (wsitools, opentile-go).
- How to add a new fixture (audit license, write LICENSE +
  PROVENANCE files, update manifest.json, rebuild tarball, upload to
  release).
- How CI consumers should integrate (the workflow snippet from §4).
- License inventory table.

## 4. wsitools integration

### 4.1 `.github/workflows/ci.yml` additions

Before the existing `go test` step:

```yaml
- name: Cache fixtures
  id: fixtures-cache
  uses: actions/cache@v4
  with:
    path: sample_files
    key: wsi-fixtures-v1-${{ hashFiles('.github/fixtures.sha256') }}

- name: Download fixtures
  if: steps.fixtures-cache.outputs.cache-hit != 'true'
  run: |
    mkdir -p sample_files
    cd sample_files
    gh release download v1 \
      --repo wsilabs/wsi-fixtures \
      --pattern '*.tar'
    for t in *.tar; do
      tar xf "$t" && rm "$t"
    done
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

- name: Verify fixture SHA-256
  run: |
    cd sample_files
    shasum -a 256 -c ../.github/fixtures.sha256
```

The existing test step uses `WSI_TOOLS_TESTDIR`:

```yaml
- name: Run tests
  env:
    WSI_TOOLS_TESTDIR: ${{ github.workspace }}/sample_files
  run: go test ./... -race -count=1
```

If today's CI doesn't already pass `WSI_TOOLS_TESTDIR`, the v0.19
PR adds it. Tests that currently `t.Skip` for missing fixture run
live afterward.

### 4.2 `.github/fixtures.sha256`

In wsitools, a checked-in file:

```
<sha256>  svs/CMU-1-Small-Region.svs
<sha256>  ndpi/CMU-1.ndpi
```

Pinned hashes. Mismatches fail the CI build (i.e. someone bumped
fixtures without updating the pin). Updating fixtures requires an
intentional wsitools PR that includes new hashes.

### 4.3 Windows compatibility

`gh` and `shasum` are available on Windows runners via the
GitHub-provided MSYS2 / Git Bash. The workflow uses `bash -l` for
the download + verify steps so Windows uses the same shell.

If `shasum` is not present on Windows runners, fallback:
`certutil -hashfile <file> SHA256` + a small inline transform.
Determined during implementation.

## 5. Licensing — initial fixtures

Both fixtures in the v0.19 initial set are **CC0** per
openslide-testdata's published license terms (third column of
https://openslide.cs.cmu.edu/download/openslide-testdata/Aperio/ and
https://openslide.cs.cmu.edu/download/openslide-testdata/Hamamatsu/).

CC0 (public domain) allows redistribution, modification, and
commercial use without attribution. Provenance is documented for
good practice but not legally required.

The `<file>.LICENSE` artifacts contain a verbatim copy of the CC0
Public Domain Dedication text (https://creativecommons.org/
publicdomain/zero/1.0/legalcode).

## 6. Test plan

### 6.1 wsi-fixtures repo

- `scripts/verify.sh` validates every fixture's SHA-256 against
  `manifest.json`. Catches accidental corruption / wrong upload.
- `scripts/pack.sh` builds the tarballs deterministically: sorted
  file list, no timestamps in tar headers. Reproducible builds for
  future audits.
- GitHub Actions in wsi-fixtures runs `verify.sh` on every push;
  catches manifest drift.

### 6.2 wsitools CI

- Existing CI matrix (macOS, Linux nocgo, Windows) continues to
  pass.
- Previously-skipped tests now run:
  - `TestConvertToDZI` (CMU-1-Small-Region.svs → DZI)
  - `TestConvertToSZI` (CMU-1-Small-Region.svs → SZI)
  - `TestConvertNDPIToDZI` (CMU-1.ndpi → DZI; ~14 s wall)
  - `TestConvertNDPIToSZI` (CMU-1.ndpi → SZI; ~18 s wall)
  - `TestConvertNDPIToCOGWSIReencode` (CMU-1.ndpi → COG-WSI)
  - `TestConvertSVSBitExact` subtest for CMU-1-Small-Region.svs

### 6.3 Acceptance gate

- Cold-cache CI runtime ≤ +90 s vs current (download + verify +
  small per-test fixture stat work).
- Warm-cache CI runtime ≤ +10 s vs current (just stat + verify).
- All fixture-gated tests pass; if any fail, the fix is part of
  v0.19 scope before tagging.

## 8. Release sequencing

Two-PR effort:

1. **`wsilabs/wsi-fixtures` initial release.** New repo + README +
   LICENSE + scripts + the two fixtures + LICENSE/PROVENANCE
   artifacts + `manifest.json` + uploaded release `v1` with
   `svs.tar`, `ndpi.tar` assets.
2. **wsitools v0.19.0.** `.github/workflows/ci.yml` update,
   `.github/fixtures.sha256`, CHANGELOG entry, version bump, tag.

Order: ship (1) first; verify the release assets are downloadable.
Then (2). wsi-fixtures has no semantic version coupling — wsitools
references `v1` directly.
